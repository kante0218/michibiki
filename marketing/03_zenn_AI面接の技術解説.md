## はじめに

AI面接——つまり、人間の面接官の代わりにAIが面接を行うシステムは、ここ数年で急速に普及しました。しかし、多くの実装は「あらかじめ用意した質問を順番に投げかけ、回答を記録する」だけの、いわば**録画面接の延長**にとどまっています。

本稿では、私が開発した AI面接プラットフォーム「[導（みちびき）](https://michibiki.tech)」で採用した**適応型質問生成アーキテクチャ**について、設計思想と技術的なアプローチを解説します。

---

## 問題定義：なぜ「固定質問」ではダメなのか

人間の優れた面接官は、候補者の回答を聞いた瞬間に次の質問を決めます。

- 回答が浅ければ、具体例を求める深掘り質問をする
- 知識に偏りがあれば、弱い領域を探る質問に切り替える
- 予想以上に詳しければ、より高度な応用問題に移行する

この**リアルタイムな判断と軌道修正**こそが、面接の質を決定づけます。固定の質問リストでは、すべての候補者に同じ質問が投げられるため、上級者には簡単すぎ、初学者には難しすぎるという問題が発生します。

## アーキテクチャ設計

導の面接システムは、以下の3層で構成されています。

### Layer 1: コンテキスト蓄積層

面接中のすべてのデータを構造化して保持します。

```
InterviewContext {
  category: "software_engineering"
  testResults: {
    multipleChoice: { score: 3/5, weakAreas: ["data_structures", "concurrency"] }
    freeForm: { score: 2/5, weakAreas: ["complexity_analysis"] }
  }
  conversationHistory: Message[]
  evaluatedSkills: Map<Skill, ConfidenceScore>
  unevaluatedSkills: Skill[]
  currentDepth: number  // 深掘りの深度
}
```

### Layer 2: 質問戦略決定層

蓄積されたコンテキストを分析し、「次に何を問うべきか」を決定します。

```
戦略決定のロジック:
1. 未評価のスキル領域があるか？
   → ある場合: そのスキルを評価する質問を生成
2. 直前の回答の深度は十分か？
   → 不十分な場合: 同じトピックで深掘り質問を生成
   → 十分な場合: 次のトピックに移行
3. テストで弱点が検出された領域か？
   → 弱点領域の場合: 基礎的な確認から始める
   → 強い領域の場合: 応用・実務レベルの質問を生成
```

### Layer 3: 質問生成層

戦略に基づいて、GPT-4で自然な質問文を生成します。ここでのポイントは、**質問の意図（何を評価したいか）と質問文（どう聞くか）を分離する**ことです。

```
// 意図と生成を分離
const intent = {
  targetSkill: "data_structures",
  evaluationGoal: "ハッシュマップの内部実装への理解度",
  depth: "intermediate",
  context: "テストで関連問題を2問不正解"
};

// GPT-4に渡すプロンプト
const prompt = `
あなたは${category}の面接官です。
以下の意図に基づいて、自然な面接質問を1つ生成してください。

評価対象: ${intent.targetSkill}
評価目的: ${intent.evaluationGoal}
難易度: ${intent.depth}
背景: ${intent.context}

制約:
- 会話の流れに自然に接続すること
- 専門用語は使いつつも、威圧的にならないこと
- 回答に2-3分かかる適切な分量であること
`;
```

## 評価フレームワーク

面接の評価は4つの軸で行います。

| 軸 | 評価内容 | 重み |
|---|---------|-----|
| 技術的正確性 | 回答が事実として正しいか | 35% |
| 深度と応用力 | 表面的な知識か、実務に適用できるレベルか | 30% |
| 構造化思考 | 論理的に整理して説明できているか | 20% |
| コミュニケーション | 簡潔で分かりやすい説明ができているか | 15% |

各軸のスコアは、面接終了後にフィードバックとして候補者に提示されます。「何が良かったか」「何を改善すべきか」を具体的に示すことで、次の面接準備に直接活かせるようにしています。

## 音声認識の工夫

リアルタイム面接では、音声認識の精度と速度が体験の質を左右します。

導では Whisper API を使用していますが、日本語特有の課題がありました。

1. **専門用語の認識精度**: 「冪等性」「計算量」「ポリモーフィズム」などの技術用語
2. **コードの読み上げ**: 変数名やメソッド名の認識
3. **フィラーの処理**: 「えーと」「あの」の適切な除去

これらは、Whisper に渡すプロンプトに分野固有の用語リストを含めることで改善しました。

## 33分野への拡張設計

導は33分野に対応していますが、分野ごとに完全に異なるシステムを構築したわけではありません。

**共通の面接エンジン**に対して、分野ごとの**設定ファイル**（評価基準、想定スキルツリー、専門用語辞書）を差し込む設計にしています。新しい分野を追加する際は、この設定ファイルを作成するだけで対応できます。
